# 完善中断机制实现

**任务时间**: 2026-03-26 00:30

## 问题描述

虽然实现了 `DynamicHumanInTheLoopMiddleware`，但前端没有收到中断信号，直接返回 `done`，用户无法进行确认操作。

## 根本原因

1. **chat_agent.py 没有检测中断状态**：
   - `agent.stream()` 在遇到中断时会正常结束（不抛出异常）
   - 中断信息保存在最后的状态中（`__interrupt__` 字段）
   - 但代码没有检查这个字段，直接发送了 `done` 事件

2. **chat.py 没有处理 interrupt 事件**：
   - 即使 `chat_agent.py` 发送了 `interrupt` 事件，路由也不会转发给前端

3. **缺少 Resume API**：
   - 用户确认后，没有 API 可以恢复执行

## 解决方案

### 1. 修改 chat_agent.py - 检测中断状态

**文件**: `app/agents/chat_agent.py`

**修改内容**:

```python
# 在 stream 循环中保存最后的状态
last_state = None

for mode, chunk in self.agent.stream(...):
    if mode == "values":
        last_state = chunk  # 保存状态
        # ...

# 在循环结束后检查中断
if last_state and '__interrupt__' in last_state:
    interrupt_info = last_state['__interrupt__']
    logger.info(f"🛑 检测到中断: {interrupt_info}")
    
    # 发送中断事件给前端
    yield {
        'type': 'interrupt',
        'interrupt_info': interrupt_info,
        'thread_id': thread_id
    }
else:
    # 发送完成事件
    yield {
        'type': 'done',
        'content': '处理完成'
    }
```

### 2. 修改 chat.py - 处理 interrupt 事件

**文件**: `app/routers/chat.py`

**修改内容**:

```python
elif event_type == 'interrupt':
    # 中断事件 - 需要用户确认
    interrupt_info = event.get('interrupt_info', {})
    thread_id = event.get('thread_id', '')
    yield f"data: {json.dumps({'type': 'interrupt', 'interrupt_info': interrupt_info, 'thread_id': thread_id}, ensure_ascii=False)}\n\n"
```

### 3. 添加 Resume API

**文件**: `app/routers/chat.py`

**新增内容**:

```python
class ResumeRequest(BaseModel):
    """恢复中断执行的请求"""
    thread_id: str
    decision: str  # "approve" 或 "reject"


@router.post("/resume")
async def resume_execution(
    request: ResumeRequest,
    chat_agent: ChatAgent = Depends(get_chat_agent)
):
    """恢复中断的执行"""
    from langgraph.types import Command
    
    # 使用 Command 恢复执行
    result = chat_agent.agent.invoke(
        Command(resume=request.decision),
        config={"configurable": {"thread_id": request.thread_id}}
    )
    
    # 返回结果
    return {
        "success": True,
        "message": content,
        "thread_id": request.thread_id
    }
```

## 工作流程

### 完整的中断-恢复流程

```
1. 用户发送消息 → 智能体开始处理
   ↓
2. 智能体调用 platform_service 工具
   ↓
3. DynamicHumanInTheLoopMiddleware.after_model 被触发
   ↓
4. 检查 doc_path 对应的文档，发现 REQUIRES_CONFIRMATION: true
   ↓
5. 动态添加 platform_service 到 interrupt_on 配置
   ↓
6. 父类 HumanInTheLoopMiddleware 检测到配置，调用 interrupt()
   ↓
7. LangGraph 暂停执行，保存状态到 checkpointer
   ↓
8. agent.stream() 正常结束，last_state 包含 __interrupt__ 字段
   ↓
9. chat_agent.py 检测到 __interrupt__，发送 interrupt 事件
   ↓
10. chat.py 转发 interrupt 事件给前端（SSE）
   ↓
11. 前端收到 interrupt 事件，显示确认对话框
   ↓
12. 用户点击"确认"或"拒绝"
   ↓
13. 前端调用 POST /chat/resume，传递 thread_id 和 decision
   ↓
14. 后端使用 Command(resume=decision) 恢复执行
   ↓
15. 智能体继续执行（如果 approve）或终止（如果 reject）
   ↓
16. 返回最终结果给前端
```

## 中断信息结构

LangGraph 的 `__interrupt__` 字段包含以下信息：

```python
{
    "__interrupt__": [
        {
            "value": "Tool execution requires approval",  # 描述
            "resumable": True,  # 是否可恢复
            "ns": ["langgraph:step:1"],  # 命名空间
            "when": "during"  # 中断时机
        }
    ]
}
```

前端可以根据这些信息：
- 显示确认对话框
- 提取工具调用信息（从之前的 tool_call 事件）
- 提供"确认"和"拒绝"按钮

## 前端实现要点

### 1. 处理 interrupt 事件

```javascript
if (event.type === 'interrupt') {
    const { interrupt_info, thread_id } = event;
    
    // 显示确认对话框
    showConfirmDialog({
        message: "此操作需要确认",
        onApprove: () => resumeExecution(thread_id, "approve"),
        onReject: () => resumeExecution(thread_id, "reject")
    });
}
```

### 2. 调用 Resume API

```javascript
async function resumeExecution(threadId, decision) {
    const response = await fetch('/chat/resume', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ thread_id: threadId, decision })
    });
    
    const result = await response.json();
    // 显示结果
    addMessage({ role: 'assistant', content: result.message });
}
```

## 技术亮点

1. **正确理解 LangGraph 中断机制**：
   - 中断不会抛出异常，而是保存在状态中
   - 需要检查 `__interrupt__` 字段来检测中断
   - 使用 `Command(resume=...)` 恢复执行

2. **完整的中断-恢复流程**：
   - 后端检测中断并通知前端
   - 前端显示确认对话框
   - 用户确认后调用 Resume API
   - 后端恢复执行并返回结果

3. **SSE 流式传输**：
   - 中断事件通过 SSE 实时推送给前端
   - 前端可以立即响应，无需轮询
