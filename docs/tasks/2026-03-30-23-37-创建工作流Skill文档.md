## 核心内容

### 工作流架构

```
workflow.json (配置)
    ↓
LangGraph (执行引擎)
    ↓
├─ 纯逻辑脚本 (数据处理、验证)
└─ 工具调用 (IT系统交互)
```

### 核心原则

1. **配置驱动**: 工作流定义在JSON中，无需修改代码
2. **脚本纯逻辑**: 脚本只处理数据，不调用外部系统
3. **工具解耦**: 外部系统调用通过platform_service工具+hook
4. **条件分支**: 支持基于状态的条件路由
5. **人工审批**: 支持关键节点的人工介入

### 节点类型

1. **script节点**: 纯逻辑脚本，处理数据、验证、计算
2. **tool节点**: 工具调用，调用platform_service访问外部系统
3. **human节点**: 人工审批，等待人工输入决策

### 边类型

1. **normal边**: 普通连接，顺序执行
2. **conditional边**: 条件分支，根据状态字段路由

### 脚本规范

```python
#!/usr/bin/env python3
import json
import sys

def main():
    # 读取输入
    inputs = json.loads(sys.stdin.read())
    
    try:
        # 纯逻辑处理
        result = process_data(inputs)
        
        # 输出结果
        print(json.dumps({
            "success": True,
            "outputs": {"result": result}
        }, ensure_ascii=False))
        
    except Exception as e:
        # 错误处理
        print(json.dumps({
            "success": False,
            "error": str(e)
        }, ensure_ascii=False))
        sys.exit(1)

if __name__ == "__main__":
    main()
```

## 最佳实践

### ✅ 推荐

1. **脚本纯逻辑**: 只处理数据，不调用外部系统
2. **工具调用**: 使用tool节点+platform_service
3. **条件分支**: 在配置中定义，不在脚本中硬编码
4. **错误处理**: 完善的try-catch和错误输出
5. **进度消息**: 每个节点定义progress消息
6. **完善文档**: 提供清晰的README.md

### ❌ 不推荐

1. **脚本耦合**: 在脚本中直接调用外部系统
2. **硬编码**: 在脚本中硬编码分支逻辑
3. **缺少错误处理**: 没有try-catch导致异常终止
4. **缺少文档**: 没有README.md让智能体无法理解
5. **缺少DOC_GUARD**: tool节点没有提供doc_path或doc_excerpt
