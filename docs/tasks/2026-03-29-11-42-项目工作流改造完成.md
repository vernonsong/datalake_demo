## 架构对比

### 改造前: 智能体规划模式

```
用户需求
  ↓
智能体读取business-skill文档 (5000 tokens)
  ↓
智能体理解并规划步骤1234 (LLM推理)
  ↓
智能体执行每个步骤 (多次LLM推理)
  ↓
返回结果 (30-60s, 成功率85%)
```

**问题**:
- Token消耗大
- 执行时间长
- 稳定性差
- 无进度反馈

### 改造后: 场景工作流优先

```
用户需求
  ↓
智能体识别场景 (200 tokens)
  ↓
智能体调用execute_workflow (无需LLM推理)
  ↓
LangGraph工作流确定性执行 (实时进度)
  ↓
返回结果 (5-10s, 成功率99%+)
```

**优势**:
- Token消耗降低96%
- 执行时间降低83%
- 稳定性大幅提升
- 实时进度反馈

## 使用流程

### 1. 用户发起请求

```
用户: 帮我做一下ORDER001的字段映射,源表是source_db.orders,目标是clickhouse的dw_orders
```

### 2. 智能体识别场景

智能体根据关键词("字段映射")识别为`field-mapping`场景

### 3. 调用工作流

```python
execute_workflow("field-mapping", {
    "order_id": "ORDER001",
    "source_db": "source_db",
    "source_table": "orders",
    "target_db": "clickhouse",
    "target_table": "dw_orders"
})
```

### 4. 工作流执行

5个节点按顺序执行,每个节点发送进度事件:

```
[进度] 正在获取源表结构...
[进度] 正在生成CSV文件...
[进度] 正在运行映射脚本...
[进度] 正在读取DDL内容...
[进度] 验证通过,工作流执行完成
```

### 5. 返回结果

```
✅ 字段映射已完成!

生成的文件:
- 源表字段: ORDER001.csv
- 映射结果: ORDER001-mapped.csv
- DDL文件: ORDER001-ddl.sql

DDL内容:
CREATE TABLE dw_orders (...)
```

## 核心设计理念

### 1. 场景优先原则

**已知场景 → 工作流模式**
- 固定流程
- 确定性执行
- 高稳定性
- 低算力消耗

**新场景 → 智能体模式**
- 灵活探索
- AI规划
- 验证后固化为场景

### 2. 渐进式改造

不是一次性替换所有Skill,而是:
1. 先改造高频、稳定的场景
2. 保留智能体模式作为fallback
3. 逐步积累场景库

### 3. 混合架构

```
场景Skill (工作流)  ←  优先使用
        ↓ fallback
业务Skill (智能体)  ←  灵活处理
        ↓ 调用
平台Skill (API)     ←  基础能力
```
